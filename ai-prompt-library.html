PROMPT
Act as an experienced technical writer with expertise in [INDUSTRY/DOMAIN].
I need to create documentation for [PRODUCT/FEATURE NAME].

Here is the context:
- Target audience: [e.g., developers, end users, IT admins]
- Documentation type: [e.g., user guide, API reference, release notes]
- Key functionality to cover: [LIST 3–5 BULLET POINTS]
- Tone: [e.g., formal, conversational, instructional]

Style Guide Compliance Report

Audit a doc against your style guide with a scored report

PROMPT
You are a documentation quality reviewer. I am giving you two inputs: (1) my team's style guide, and (2) a document I have written. Your task is to audit the document against the style guide and produce a detailed compliance report.

STYLE GUIDE:
[PASTE YOUR FULL STYLE GUIDE HERE — or paste the relevant sections]

DOCUMENT TO REVIEW:
[PASTE YOUR DOCUMENT HERE]

Produce a report structured as follows:

## Compliance Report: [Document Title]

### 1. Overall Score
Give an overall compliance score out of 100, with a 2-sentence summary.

### 2. Category Scores (score each out of 10)
- Voice and Tone
- Formatting and Structure
- Terminology Consistency
- Sentence Clarity and Length
- Use of Headings and Lists
- Grammar and Punctuation

### 3. Violations Found
For each violation: quote the original text, state which style guide rule it breaks, and provide a corrected version.

### 4. Positive Observations
List 3–5 things the document does well against the style guide.

### 5. Priority Action List
Rank all violations as CRITICAL / MAJOR / MINOR and list them in order of priority.
Pro Tip: If your style guide is long, paste only the most relevant sections (Voice & Tone + Formatting) to keep the audit focused.

Terminology & Consistency Audit

Find inconsistent terms, caps, hyphens, and abbreviations

PROMPT
You are a technical editor specializing in terminology management. Audit the following document for terminology and consistency issues.

Document:
[PASTE DOCUMENT HERE]

Preferred terminology list (if available):
[PASTE YOUR GLOSSARY OR PREFERRED TERMS — or write "None provided"]

Perform the following analysis and produce a structured report:

1. TERMINOLOGY INCONSISTENCIES
   List every case where the same concept is referred to by different names (e.g., "end user" vs "user" vs "customer"). For each: show all variants found, their locations, and the recommended single term to use.

2. CAPITALIZATION INCONSISTENCIES
   Flag any terms that are capitalized in some places and lowercase in others.

3. HYPHENATION INCONSISTENCIES
   Flag terms that appear both hyphenated and unhyphenated.

4. ABBREVIATION ISSUES
   Find abbreviations used without being defined on first use, or defined more than once.

5. RECOMMENDED GLOSSARY
   Based on your findings, suggest a 10-term mini glossary with preferred spellings and definitions.
Pro Tip: Run this before a major release. Save the recommended glossary output as your official terminology reference going forward.
10

Readability & Accessibility Check

Score and improve for plain language and accessibility

PROMPT
You are a plain language and accessibility specialist. Analyze the following document and produce an accessibility and readability report.

Document:
[PASTE DOCUMENT HERE]

Target reading level: [e.g., Grade 8, general public, technical professional]

Provide the following:

1. READABILITY ASSESSMENT
   Estimate the current reading level (Flesch-Kincaid equivalent). List the 5 most complex sentences and rewrite each in plain language.

2. PLAIN LANGUAGE AUDIT
   Identify: passive voice instances, nominalizations (e.g., "make a decision" → "decide"), overly long sentences (20+ words), and weak openers (e.g., "There are", "It is").

3. ACCESSIBILITY CHECKLIST
   Evaluate against these criteria (Pass / Needs Work / Fail):
   - Logical heading hierarchy (H1 > H2 > H3)
   - Alt text guidance for images (are images described in surrounding text?)
   - Link text quality (e.g., "click here" vs descriptive links)
   - Use of color or formatting as the sole indicator of meaning
   - Consistent list formatting

4. REWRITTEN EXCERPT
   Rewrite the most problematic paragraph using plain language principles.

5. SUMMARY SCORE
   Give an overall accessibility readiness score (1–10) with a brief rationale.
Pro Tip: Use this before publishing any customer-facing documentation. The before/after excerpt is a great stakeholder artifact.

Demo Video → How-To Guide

Turn a video transcript into step-by-step instructions

PROMPT
You are a technical writer. I have a transcript from a product demo video and I need you to turn it into polished, structured documentation.

Product being demonstrated: [PRODUCT/FEATURE NAME]
Target audience: [e.g., new users, IT administrators, developers]
Output document type: [e.g., step-by-step how-to guide, feature overview, getting started guide]

Video transcript:
[PASTE THE FULL VIDEO TRANSCRIPT HERE — raw auto-captions are fine]

Instructions:
- Ignore filler words, false starts, and off-topic commentary
- Extract every distinct step or action the presenter performs and turn them into numbered instructions
- Where the presenter says "as you can see here" or refers to something visual, replace with a written description of what should be visible and add a placeholder: [SCREENSHOT: describe what to capture]
- Group related steps under descriptive headings
- Add a Prerequisites section at the top for anything mentioned as a setup requirement
- Add a "What you'll achieve" intro paragraph
- Flag any part of the transcript that is unclear or contradictory with [VERIFY WITH SME]
- End with a list of follow-up questions to ask the presenter
Pro Tip: Use YouTube auto-captions, Otter.ai, or Whisper to get a rough transcript first. Even messy transcripts produce great results.

Convert Documentation to Markdown

Reformat any doc into clean, publish-ready Markdown

PROMPT
You are a technical writer and documentation engineer. Convert the following document into clean, well-structured Markdown.

Document to convert:
[PASTE YOUR DOCUMENT HERE]

Markdown flavor: [e.g., standard Markdown / GitHub Flavored Markdown (GFM) / MDX / MkDocs / Docusaurus]
Where this will be published: [e.g., GitHub README, Confluence, internal wiki, docs site]

Conversion rules to follow:
- Use ATX-style headings (# H1, ## H2, ### H3) — never use underline-style
- Convert all numbered lists to proper Markdown ordered lists
- Convert all bullet points to unordered lists (-)
- Format all code, commands, file paths, and UI elements in backtick code spans
- Format multi-line code blocks with triple backticks and a language identifier (e.g., \`\`\`bash, \`\`\`json)
- Convert any tables to Markdown pipe tables with header row and alignment separators
- Replace bold/italic formatting with **bold** and _italic_ syntax
- Add a blank line before and after every heading, list, and code block
- Preserve all heading hierarchy from the original
- Add a document-level front matter block at the top: title, description, and date fields (leave values as placeholders if unknown)

After the converted document, include:
1. A short summary of conversion decisions made
2. A list of any elements that could not be converted cleanly, with recommendations
Pro Tip: Specify the Markdown flavor — GFM supports tables and task lists that standard Markdown doesn't.

Convert Documentation to DITA XML

Transform prose into structured DITA topics with a map

PROMPT
You are a structured authoring specialist and technical writer experienced in DITA XML. Convert the following document into valid DITA XML.

Document to convert:
[PASTE YOUR DOCUMENT HERE]

DITA version: [e.g., DITA 1.3 / DITA 2.0]
Topic type: [e.g., concept / task / reference / troubleshooting — or "choose the best type for each section"]
Toolchain/output: [e.g., Oxygen XML, DITA-OT, Heretto, Paligo — or "generic DITA"]

Conversion requirements:
- Split the document into separate DITA topics as appropriate (one topic per logical unit of information)
- Use correct topic type elements: <concept>, <task>, or <reference>
- In task topics: wrap each step in <step><cmd>...</cmd></step> inside <steps>
- In concept topics: use <conbody> with <p>, <ul>, <ol>, and <note> elements
- In reference topics: use <refbody> with <table> or <properties> elements for structured data
- Use <shortdesc> for every topic (1–2 sentence description immediately after <title>)
- Mark up UI elements with <uicontrol>, code with <codeph>, file paths with <filepath>, and variables with <varname>
- Use <note type="warning"> and <note type="tip"> appropriately
- Wrap the output in a DITA map (<map>) that references all generated topics

After the XML output, provide:
1. A topic list showing the hierarchy of generated topics
2. Notes on any content that required interpretation or restructuring
Pro Tip: Always validate the output in Oxygen XML or a DITA linter before committing to your toolchain.

Convert Documentation to DocBook XML

Reformat content into DocBook 5 with semantic elements

PROMPT
You are a technical writer and XML authoring specialist. Convert the following document into valid DocBook 5 XML.

Document to convert:
[PASTE YOUR DOCUMENT HERE]

DocBook output type: [e.g., <article>, <book>, <chapter>, <section>]
Publishing tool: [e.g., Publican, Asciidoctor with DocBook backend, generic DocBook — or "not specified"]

Conversion requirements:
- Use DocBook 5 namespace: xmlns="http://docbook.org/ns/docbook" version="5.0"
- Wrap the document in the appropriate root element (<article> or <book>)
- Use <section> with <title> for each major heading
- Convert procedural content into <procedure><step><para>...</para></step></procedure>
- Mark up: <command> for CLI commands, <filename> for file paths, <literal> for code values, <guilabel> for UI elements, <keycap> for keyboard keys
- Use <itemizedlist> and <orderedlist> for bullets and numbered lists
- Use <table> with <thead> and <tbody> for any tabular data
- Use <note>, <warning>, <tip>, <caution> elements for callouts
- Add <info> block at the top with <title>, <author> (placeholder), and <pubdate> (placeholder)
- Add xml:id attributes to all <section> elements using kebab-case slugs

After the XML, list any DocBook semantic elements you chose and why.
Pro Tip: Specify your publishing tool — Publican and Asciidoctor have slightly different expectations for structure and ID conventions.

Adapt Doc for a Different Output Channel

Single-source one doc to print, mobile, chatbot, or in-app

PROMPT
You are a technical writer specializing in single-sourcing and multi-channel publishing. I have an existing document written for [ORIGINAL FORMAT, e.g., printed user manual / web help center] and I need to adapt it for [TARGET FORMAT, e.g., in-app contextual help / mobile-first help article / PDF quick reference card / slideshow / chatbot FAQ].

Original document:
[PASTE DOCUMENT HERE]

Target channel constraints:
- Maximum length: [e.g., 300 words, 1 screen, 5 slides]
- Reading context: [e.g., user is mid-task and needs quick answers, not background reading]
- Format limitations: [e.g., no tables, no images, plain text only / HTML supported / Markdown only]

Adapt the document by:
1. Cutting all content that doesn't serve a user who is actively trying to complete a task
2. Rewriting for the target context (e.g., shorter sentences for mobile, imperative voice for in-app)
3. Restructuring if needed (e.g., convert prose to Q&A for chatbot, convert steps to a checklist for quick ref)
4. Flagging any content cut that may need a separate, deeper article

Output: the adapted document + a brief editorial note explaining key decisions made.
Pro Tip: Describe the user's mental state at the point of reading — 'frustrated and mid-task' leads to very different output than 'onboarding'.

Create a Troubleshooting Guide

Structured entries to help users self-resolve common issues

PROMPT
Act as a technical writer creating a troubleshooting guide for [PRODUCT/SYSTEM NAME].

Target audience: [e.g., end users, IT support staff]
Support context: [e.g., standalone guide, embedded in help center, printed handout]

Known issues to cover:
[LIST 3–6 ISSUES OR SYMPTOMS — e.g., "Cannot log in", "Sync fails", "Dashboard loads slowly"]

For each issue, write a troubleshooting entry that includes:
1. Issue title — written as the user would describe it (e.g., "I can't log in")
2. Symptoms — what the user sees or experiences
3. Possible causes — brief list
4. Resolution steps — numbered, with clear actions
5. If unresolved — escalation path or next step

Use plain language. Avoid blaming the user. Where helpful, include a note if the issue is a known bug with a workaround.
Pro Tip: Pull issue titles directly from support tickets or user feedback — this makes the guide immediately relevant to real users.

Build a Documentation Plan

Scope, deliverables, timeline, and risks for a product launch

PROMPT
You are a lead technical writer and documentation strategist. Create a comprehensive documentation plan for a new product launch.

Product name: [PRODUCT NAME]
Product type: [e.g., SaaS platform, developer tool, hardware device, internal enterprise tool]
Target audiences: [LIST ALL AUDIENCES — e.g., end users, IT admins, developers, support agents]
Launch date: [DATE OR TIMEFRAME]
Team size: [e.g., 1 writer, 3 writers, embedded in product squads]
Known features to document: [LIST KEY FEATURES]

Produce a documentation plan that includes:

1. DOCUMENTATION SCOPE
   List every deliverable needed at launch vs. post-launch (as a prioritized table: Deliverable | Audience | Priority | Owner | Format).

2. CONTENT HIERARCHY
   Propose a recommended information architecture — the top-level structure of the documentation site or help center.

3. WRITING TIMELINE
   A phased schedule from first draft to publication, including review cycles. Flag dependencies (e.g., "needs UI freeze before screenshots").

4. RESOURCE ESTIMATE
   Rough estimate of writing effort per deliverable (hours or days). Flag which items are highest-risk due to complexity or dependency on SME access.

5. RISKS & MITIGATIONS
   List 3–5 risks (e.g., late feature changes, SME unavailability) and a mitigation plan for each.

6. SUCCESS METRICS
   Suggest 3–5 measurable outcomes to define whether the documentation is successful post-launch.
Pro Tip: Use the Resource Estimate output to make a business case for headcount or contractor support.

Prompt Engineering Principles

Every prompt in this library is built on five core principles that make them work across all major AI platforms.

Role Assignment

Opens with "Act as..." or "You are..." to prime the model with the right expertise and tone.

Structured Context

Provides all background — audience, purpose, format — before stating the task, reducing guesswork.

Explicit Output Format

Specifies exactly what sections, structure, or format the response should follow.

Bracketed Placeholders

Every customizable field is marked [LIKE THIS] so nothing is accidentally submitted as-is.

Iteration Guidance

Pro Tips tell you how to follow up or adapt each prompt for your specific context.
